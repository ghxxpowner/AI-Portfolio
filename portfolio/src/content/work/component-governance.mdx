## Context

The company had a component library, a Figma kit, and a problem: both were maintained by whoever had spare time, which meant neither was maintained. Product squads had quietly stopped waiting — 59% of new UI shipped as local one-offs, and the library's issue tracker had become a graveyard of polite requests. I joined to run the system as a product, with a team of four serving twenty-three squads.

## Problem

Four people cannot build components for twenty-three squads; the math only works if the squads build and the system team governs. But the previous "everyone can contribute" policy had failed — contributions sat unreviewed for weeks, so people stopped contributing, so the library fell behind, so people built one-offs. The actual design problem was a **contribution process fast enough that using the system is lazier than bypassing it**.

## My role

I designed the governance model end to end — the contribution tiers, the RFC template, the review SLA, and the component lifecycle states — and ran it for the first year: triaging proposals, chairing reviews, and reporting adoption to leadership. My team owned quality gates; squads owned building.

## Process

### One front door, three sizes of yes

Every request now enters through a single intake with a 48-hour triage SLA, and gets routed by weight: a *patch* (token fix, prop addition) needs one system reviewer; an *extension* (new variant of an existing component) needs a spec review; only a genuinely *new component* needs an RFC and a working session. The previous process treated a border-radius fix and a new data table with the same ceremony — which is why both took eleven days.

### A lifecycle that lets things be imperfect in public

Components now carry an explicit state — *experimental*, *stable*, *deprecated* — surfaced in both Figma and Storybook. Squads can publish an experimental component into a shared namespace immediately; promotion to stable is where the full quality gate (accessibility audit, API review, docs) applies. This killed the worst dynamic of the old model, where work hid in product codebases because the bar for entering the library was the bar for perfection.

### Making adoption visible

We instrumented the codebase: a weekly report counts system components versus local one-offs per squad, and the trend — not the absolute number — goes to each squad's engineering lead. No mandates were issued. Squads near the bottom of the chart asked for migration help within weeks; being measurably behind your peers turned out to be the only enforcement mechanism we needed.

## Decisions & trade-offs

<Callout label="Decision">
  Review capacity is **scheduled, not heroic**: every system team member has two fixed review slots per week, and intake throttles to match. Predictable three-day turnarounds beat occasional same-day ones — contributors plan around an SLA, not around hope.
</Callout>

<Callout label="Trade-off">
  The experimental namespace means **the library contains components we wouldn't have designed**. That's the cost of contribution at scale, and we accepted it deliberately: a federated system you can influence beats a pristine one nobody uses. The lifecycle states make the quality difference legible instead of pretending it away.
</Callout>

## Outcomes

A year in, system coverage of product UI rose from 41% to 87%, and — the number I'm proudest of — 64% of new components were contributed by product squads, not built by my team. The system stopped being a bottleneck with a backlog and became infrastructure with a queue measured in days.

## Reflection

I'd publish the adoption dashboard from week one. We spent months making the contribution process good before making the gap visible, and visibility turned out to be what created demand for the process. The order should have been reversed.
